Shared WebAssembly.Memory reclamation test

Validates the hinge behind the Safari image-switch OOM fix: does Worker.terminate() promptly reclaim a large committed shared WebAssembly.Memory owned only by workers, while a memory the persistent main thread co-owns is not reclaimed? Each iteration allocates a fresh 1 GiB-maximum shared memory, grows it to the committed size, and touches every page (real physical commit). If terminate() reclaims, live memory stays flat and every iteration completes. If it does not, committed memory accumulates until the tab throws Out of Memory — the same error users hit when switching images.

Parameters

A · Worker-exclusive

Worker allocates & owns the memory. Main thread never receives it. Expected: flat, all iters complete.

B · Main-thread co-owned

Worker ships the memory to the main thread, which keeps it (the leak shape). Expected: OOM after a few iters.

C · Worker↔worker shared

Memory shared between a worker and its sub-worker; both terminated, main thread never holds it. Expected: flat, like A.
